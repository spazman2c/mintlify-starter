# Promise Date Resend

Use this runbook when SO promise date changes were entered in IQ Reseller but the external partner did not receive the updated ETA through the MRP/Symphony interface.

The promise date field is read from `st.deliverydate`. IQ Connect sends it in the `voupdates` XML feed as `ExpectedActivityDate`.

## Confirm the affected SOs

The customer usually provides partner PO numbers. Map those to IQ SOIDs first:

```sql
SELECT
    q.soid,
    q.mktsoid AS [Partner_PO],
    q.custsoid,
    q.status AS iqconnect_status,
    q.eta AS [Last_ETA_Sent_By_IQConnect],
    s.deliverydate AS [Current_SO_Promise_Date],
    s.status AS so_status,
    q.datecreatedutc
FROM iqconnect_so q
LEFT JOIN st s
    ON s.stid = q.soid
WHERE q.mktsoid IN ('<partner-po-1>', '<partner-po-2>')
ORDER BY q.mktsoid;
```

The normal scheduled `soupdate` job scans a limited status set. If the promise date changed but the row is no longer eligible for the normal scan, use the resend table and manual command below.

## Queue the resend rows

Insert only the affected IQ SOIDs into `iqconnect_batch_soupdate_resend`:

```sql
INSERT INTO iqconnect_batch_soupdate_resend (soid)
VALUES
(<soid-1>),
(<soid-2>);
```

Confirm the rows are queued:

```sql
SELECT *
FROM iqconnect_batch_soupdate_resend;
```

## Run the manual resend

Open Kudu for the App Service:

```text
Azure Portal -> <app-service-name> -> Development Tools -> Advanced Tools -> Go -> Debug console -> CMD
```

If you do not know the exact WebJob folder, locate `IqSync.exe` first:

```cmd
cd /d D:\home\site\wwwroot
dir /s /b IqSync.exe
```

Run the resend from Kudu CMD using the matching Symphony WebJob executable:

```cmd
"D:\home\site\wwwroot\App_Data\jobs\common\<webjob-folder>\IqSync.exe" to-bestbuy resendsoupdate <tenant-key> production delta
```

## Verify completion

After the command finishes, the resend table should be empty for processed SOIDs:

```sql
SELECT *
FROM iqconnect_batch_soupdate_resend;
```

Then confirm a new `*_voupdates.xml` file was created and sent through the configured Symphony output path.

If `iqconnect_so.eta` already equals `st.deliverydate`, IQ Connect may treat the ETA as already sent. Confirm whether the partner actually received the file before changing data.

## Why updates can be missed

The common failure pattern is not that the promise date field failed to save. The promise date can be present on the SO while the scheduled `soupdate` scan skips the row because its `iqconnect_so.status` or SO status is outside the normal eligibility window. The manual resend queue bypasses that scan by explicitly selecting the SOIDs that need a new `voupdates` file.
